Domain

Non-repudiation

Non-repudiation is the simplest DAR win: durable proof that a high-stakes action occurred—who did it, what they did, when it committed, and under which referenced context—without relying on internal logs.

What breaks without DAR
After an incident or dispute, parties argue about approvals: “I didn’t approve that,” “the system changed,” “the policy was different.” DAR anchors a stable evidence artifact at action time.

Typical actions

  • loan approvals / credit decisions
  • wire approvals / payment authorizations
  • policy exceptions / overrides
  • privileged admin changes
  • contract acceptance / consent capture
prove approval bind context portable evidence

Receipt contents (minimal)

  • actor, action, object, timestamp
  • receipt_ref (correlation key)
  • policy_version_ref (and optional config_ref)
  • integrity proof (signature / hash chain)

Store references—avoid storing sensitive payloads.

Event taxonomy examples

approval.requested approval.granted approval.denied approval.overridden exception.granted consent.captured

Keep the taxonomy stable. You can map internal system events to this interface.

Demo workflow (recommended)

  • UI collects approval intent (front-end)
  • Back-end commits approval + conditions
  • Receipt emitted post-commit (non-blocking)
  • Receipt stored immutably in customer cloud
  • Later: audit/dispute timeline verifies receipts

Storage & custody

Receipts are written into customer-controlled immutable storage (DAR is not a system of record).

  • AWS S3 Object Lock
  • Azure Immutable Blob
  • GCP Retention Bucket

One line DAR mapping

Non-repudiation = a receipt that proves a specific approval/authorization occurred at a specific time, under referenced policy/config context, with integrity proof.
