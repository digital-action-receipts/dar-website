Domain

Audit continuity

Audits ask for proof that survives time, change, and vendor turnover. DAR provides portable evidence: receipts issued at action time, held under customer custody, independently verifiable later.

What breaks without DAR
Audit evidence often depends on the vendor’s database, internal logs, or screenshots. When systems migrate or vendors change, evidence becomes hard to verify. Receipts remain valid even when systems evolve.

Audit evidence DAR supports

  • approval controls and exceptions
  • change management timelines
  • policy/version traceability
  • AI inference traceability (referenced)
  • transaction execution proof

Receipt characteristics auditors like

  • immutable / append-only storage
  • consistent taxonomy
  • independent verification path
  • clear chain-of-custody (customer custody)
  • compact evidence (references, not payloads)

Event taxonomy examples

control.attestation.recorded approval.granted exception.granted policy.published config.changed deploy.promoted transaction.executed

Demo workflow (recommended)

  • Define a small set of control events
  • Emit receipts post-commit for each event
  • Store receipts in customer immutable storage
  • Provide “audit timeline view” that verifies receipts
  • Keep registries separate (policy/model/config)

Storage & custody

Customer-controlled immutable storage (DAR is not a system of record).

  • AWS S3 Object Lock
  • Azure Immutable Blob
  • GCP Retention Bucket
