Domain

Access decisions

Access disputes often hinge on a single question: “Who had access, and why?” DAR provides durable proof of grants, denials, role changes, and privileged actions—without becoming a monitoring system.

What breaks without DAR
After an incident, teams argue about entitlements: “the user never had that role,” “the policy wasn’t in effect,” “the audit log is incomplete.” Receipts anchor the decision and referenced context at the moment it commits.

Typical actions

  • role granted / revoked
  • entitlement change approved
  • privileged session started / ended
  • policy exception granted
  • access request approved / denied

Receipt contents (minimal)

  • actor (requestor + approver if applicable)
  • action (grant/deny/revoke)
  • object (resource + principal)
  • timestamp + receipt_ref
  • policy_version_ref + optional config_ref
  • integrity proof

Receipts store references—avoid storing secrets or session contents.

Event taxonomy examples

access.requested access.granted access.denied access.revoked role.assigned role.removed privileged.session.started privileged.session.ended

Demo workflow (recommended)

  • User requests access
  • Approval commits in IAM/workflow system
  • Receipt emitted post-commit (non-blocking)
  • Receipt stored immutably in customer cloud
  • Later: incident response verifies entitlements timeline

Storage & custody

Customer-controlled immutable storage (DAR is not a system of record).

  • AWS S3 Object Lock
  • Azure Immutable Blob
  • GCP Retention Bucket
