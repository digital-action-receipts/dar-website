DAR (Open primitive)

DAR is the minimal, portable receipt format: proof an action occurred — not a judgment of correctness or compliance.

Minimal example

Small by design

Example below includes a separate envelope (signature / receipt hash / locator). Canonicalization and hashing are performed over the payload only. 

{
  "payload": {
    "dar_version": "1.0",
    "action_type": "ai_decision",
    "action_id": "issuer:billing-bot-v3:evt_0192",
    "timestamp": "2026-01-10T02:15:00Z",
    "actor": {
      "type": "agent",
      "id": "billing-bot-v3"
    },
    "inputs_hash": "sha256:abc123...",
    "outcome_hash": "sha256:def456..."
  },
  "receipt_hash": "sha256:0000...",
  "signature": {
    "algorithm": "ed25519",
    "key_id": "kid:billing-bot-v3",
    "value": "BASE64..."
  },
  "locator": "s3://customer-bucket/dar/sha256-0000.json"
}

Principle: keep payload minimal; store references (hashes/IDs) — not sensitive data.

Why now

As autonomous systems and AI-driven workflows grow, internal logs don’t translate across vendors, customers, and auditors. DAR makes actions portable; SV-DAR strengthens trust when incentives conflict.

Portability

Evidence should travel

When workflows span tools and vendors, a shared proof format reduces ambiguity and friction.

Accountability

Receipts make it easier to reconstruct what happened without granting special access to internal systems.

Disputes

Proof must survive scrutiny

SV-DAR targets the workflows where evidence must remain defensible under dispute and audit.
